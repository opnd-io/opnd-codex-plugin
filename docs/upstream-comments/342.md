Same root cause as #281 — broker busy / timeout swallowed as `loggedIn: false`. Our fork [`opnd-io/opnd-codex-plugin`](https://github.com/opnd-io/opnd-codex-plugin) addresses both in PR #4 commit [`3d72d6e`](https://github.com/opnd-io/opnd-codex-plugin/commit/3d72d6e).

## Fix location

`plugins/opnd-codex/scripts/lib/codex.mjs` `getCodexAuthStatusFromClient` catch now branches:

- `error.rpcCode === BROKER_BUSY_RPC_CODE` (-32001) → returns `loggedIn: null + transient: true` (broker is busy, actual auth state unknown)
- regex `/timed out|timeout|ECONNRESET|EPIPE/i` → same transient branch (broker stuck case — large SQLite WAL flush etc.)

The new `transient` field lets `buildSetupReport` (and downstream `parseSetupJson` in the daily-evolve health-check) distinguish *transient broker contention* from *actual logged-out* state.

`auth-health-check.mjs` adds a new `HEALTH_STATUS.TRANSIENT` enum with `decideDegrade(TRANSIENT → PROCEED)` so routines don't degrade to heuristic fallback when the broker is merely busy.

## Direct-fallback option (scope = `getCodexAuthStatus` only)

The fork hasn't added a direct app-server spawn fallback specifically for the `getCodexAuthStatus` busy-broker path — it still surfaces `transient: true` + recovery hint instead of retrying. If upstream prefers `getCodexAuthStatus` to retry with a fresh direct spawn on `BROKER_BUSY_RPC_CODE`, the fix could land at the `restartStaleBrokerSession()` site already used for `#41` stale-auth (similar lock-bounded restart, with caller-bounded retry cap to avoid loops). Scope: limited to setup/auth-status probe path; non-auth turn/start paths already have separate handling via `withModelFallback` and `withAppServer`'s direct-spawn branch for `--profile`/`--fast`.
