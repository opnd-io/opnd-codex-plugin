Our fork [`opnd-io/opnd-codex-plugin`](https://github.com/opnd-io/opnd-codex-plugin) addresses this with the **cd-first hook pattern**:

## Pattern

All Claude Code hook scripts that reference plugin paths use:

```bash
cd "${CLAUDE_PLUGIN_ROOT}" && node scripts/path/to/hook.mjs
```

instead of `node "${CLAUDE_PLUGIN_ROOT}/scripts/path/to/hook.mjs"`. On Windows when CWD is on a different drive (e.g., session on `D:` while plugin is on `C:`), the direct-path invocation can fail to locate the script itself (some shell + Node path-resolution combinations don't traverse the cross-drive absolute path correctly), or — when the script *does* launch — relative `import` / `require` paths inside the script resolve against the wrong drive and silently break. `cd` first forces the subprocess to start in the plugin's drive, which makes both path resolutions deterministic.

See [`plugins/opnd-codex/hooks/hooks.json`](https://github.com/opnd-io/opnd-codex-plugin/blob/main/plugins/opnd-codex/hooks/hooks.json) for the applied pattern across SessionStart / SessionEnd / Stop hooks.
